# Exercise 1: Write Your OVERVIEW.md

> Before you write a single line of code, describe what you're building. This is the document you'd hand to someone who asks "so what does your agent do?"

## What is OVERVIEW.md?

An overview document answers three questions:
1. **What** does this agent do?
2. **Who** is it for?
3. **Why** does it exist? (What problem does it solve?)

This is not a technical document. It's a plain-language description that anyone on your team (or outside it) can read and immediately understand.

## Instructions

Create a file called `OVERVIEW.md` in your project root. It should include:

### 1. Agent name and one-liner
Pick a name for your agent. Write a single sentence that describes what it does.

Example:
> **InboxPilot** — An agent that triages your email inbox by urgency and drafts response suggestions.

### 2. The problem
Describe the problem your agent solves. Be specific. What's painful about the current way of doing this? Why would someone want an agent for this instead of doing it manually?

2-3 sentences max. If you can't describe the problem concisely, you might not understand it well enough yet.

### 3. How it works (non-technical)
Describe the flow in plain language. What does the agent do step by step, from the user's perspective? No code, no implementation details. Just: "You give it X, it does Y, you get Z."

### 4. What it produces
What's the output? A report? A cleaned dataset? A set of recommendations? A file? Be concrete.

### 5. Scope and limitations
What does the agent NOT do? Where are the boundaries? This is just as important as what it does. Defining scope prevents your agent from becoming a sprawling mess.

## Quality check

Before moving on, read your OVERVIEW.md and ask:
- [ ] Could someone with zero technical knowledge understand what this agent does?
- [ ] Is the problem statement specific, not generic?
- [ ] Are the scope boundaries clear?
- [ ] Does the one-liner actually describe the agent (not just the technology)?

## Time

~10 minutes. Keep it short and focused. You'll refine it as you build.
